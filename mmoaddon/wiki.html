Overview

SLMMOStatsApply is an addon plugin for MMOItems and MythicLib that introduces a Path of Exile 2–inspired modifier system. Players can drag and drop "rune" items onto their gear to permanently apply stat bonuses.

The stats come directly from the MMOItem rune itself — the plugin reads whatever stats the server admin configured on the rune and merges them into the target item. This gives admins full control over what each modifier does without editing plugin config files.

Key features:

  • Drag-and-drop modifier application via inventory clicks
  • 5-tier system with configurable success rates per tier
  • Prefix/suffix naming system (e.g. "Sharp Diamond Sword of Fortification")
  • Conflict and synergy rules between modifiers
  • Critical success chance for doubled stats
  • Rune abilities/skills transfer (copied from rune to target)
  • POE2-style lore display: aggregated stats + compact rune list
  • Fully configurable stat formats and display order
  • Protection and removal scrolls
  • Modifier crafting: combine 2x same tier into 1x next tier
  • Configurable animations (particles, sounds, titles)
  • Per-tier modifier limits based on MMOItems item tier
  • PlaceholderAPI support

Installation

  1. Place SLMMOStatsApply-1.0.0-all.jar into your plugins/ folder.
  2. Make sure MMOItems and MythicLib are installed.
  3. Restart the server. The plugin will generate all config files.
  4. Create your rune items in MMOItems (see Creating Runes).
  5. Define modifier metadata in modifiers.yml.

How It Works

The modifier system follows a simple loop:

  1. Admin creates a rune item in MMOItems with specific stats (attack damage, defense, etc.).
  2. Admin adds three custom stats to that rune: APPLICABLE: true, MODIFIER_ID, and MODIFIER_TIER.
  3. Player picks up the rune on their cursor.
  4. Player clicks on a target MMOItem in their inventory.
  5. The plugin validates the action (type restrictions, max slots, conflicts, permissions).
  6. A success roll is made based on the tier's configured success rate.
  7. On success: all stats from the rune are read and merged into the target. On failure: the rune is consumed (optionally).
  8. The item's lore and name update automatically.
Note Stats are never defined in modifiers.yml. The plugin reads stats directly from the rune's NBT data at the moment of application. Whatever stats you put on the MMOItem rune — those are the stats that get applied. Abilities (skills) are also transferred from rune to target item.

Lore Display (POE2 Style)

Inspired by Path of Exile 2, applied modifiers are not shown individually. Instead, the item lore displays an aggregated summary of all stat bonuses from all applied runes, followed by a compact list of applied runes.

Example Output

— Existing MMOItems lore above —

═══ Rune Bonuses (3/6) ═══
  ➸ Attack Damage: 100
  ■ Crit Strike Chance: +30%
  🔥 Fire Damage: +5
  | Right Click ||| Firebolt
─── Runes ───

Abilities / Skills

Rune items can also carry MMOItems abilities (skills). When a rune is applied to a target item, all abilities from the rune are transferred and appended to the target item's ability slots.

How It Works

  1. Create a rune in MMOItems and assign abilities to it using the MMOItems GUI editor (e.g. Firebolt, Heal, Frostbolt).
  2. When the rune is applied to a target item, the abilities are read from the rune's NBT tags (MMOITEMS_ABILITY_0, MMOITEMS_ABILITY_1, etc.).
  3. The abilities are appended to the target item's next available ability slots.
  4. When a modifier is removed (via scroll or command), its abilities are also removed and the remaining abilities are compacted.

Lore Display

Abilities appear in the aggregated stats section with the format defined in stat-formats.yml under the ability section. Available placeholders:

Placeholder Description
{ability} Ability display name (e.g. Firebolt)
{trigger} Trigger type (Right Click, Left Click, Sneak, etc.)
{modifier} Modifier name (e.g. Level, Cooldown, Damage)
{value} Modifier value
Note Ability modifiers (Level, Cooldown, Mana Cost, Damage, etc.) are parsed from the ability's JSON data and displayed using the modifier-foreach format with modifier-splitter between them.

Creating Runes in MMOItems

To create a modifier rune, make an MMOItem with these three custom stats set on the item:

Stat Type Description
APPLICABLE Boolean Must be true for the item to be recognized as a modifier rune.
MODIFIER_ID String Must match a key in modifiers.yml (e.g. sharp).
MODIFIER_TIER Number The tier level (1–5).

Then set whatever stats you want the rune to grant. Example MMOItems config: 

SHARP_RUNE_T3:
  base:
    material: PAPER
    name: '&fSharp Rune III'
    attack-damage:
      base: 7
      scale: 0.3
      spread: 0.1
      max-spread: 0.3
    critical-strike-chance: 30.0
    lore:
      - "&7Drag onto a weapon to apply"
      - "&7+7 Attack Damage"
    # --- Required SLMMOStatsApply stats ---
    APPLICABLE: true
    MODIFIER_ID: sharp
    MODIFIER_TIER: 3

When this rune is dragged onto a weapon, the plugin reads the attack-damage and critical-strike-chance values (with scale/spread applied by MMOItems) and adds them to the target.

Applying Modifiers

Drag and Drop

Pick up a rune item on your cursor and click it onto a target MMOItem in your inventory. The plugin intercepts the click and processes the modifier application.

Success and Failure

Each tier has a success rate configured in modifiers.yml. Higher tiers have lower success rates. When application fails:

  • The rune is consumed if apply.consume-on-fail is true.
  • If the player has a Protection Scroll in their inventory, it is consumed instead and the target item is saved.

Critical Success

Each modifier has a critical-chance value. When a critical success occurs, all stats from the rune are multiplied by the critical-multiplier (default 2.0x) before being applied.

Tier System

Modifiers support up to 5 tiers. Each tier defines:

  • success-rate — percent chance for the apply to succeed
  • weight — used for weighted random selection (crafting recipes)

Example from the default "sharp" modifier:

Tier Roman Success Rate Weight
1 I 95% 1
2 II 85% 2
3 III 70% 3
4 IV 50% 4
5 V 30% 5

Prefix / Suffix

Each modifier is either a PREFIX or a SUFFIX. This affects:

  • Item name — prefixes appear before the original name, suffixes after. For example: "Sharp Diamond Sword of Fortification"
  • Slot limits — prefixes and suffixes have separate max counts per item.
Type Example Name Format
Prefix Sharp, Deadly, Blazing {name} Original
Suffix of Fortification, of Luck Original {name}

The name format is controlled by name-transform.format in config.yml.

Conflicts

Modifiers can define a list of conflicting modifier IDs. If a target item already has a conflicting modifier, the new one cannot be applied. 

blazing:
  conflicts: [frozen]    # Cannot coexist with Frozen

frozen:
  conflicts: [blazing]   # Cannot coexist with Blazing

Conflicts are checked bidirectionally — if A conflicts with B, it doesn't matter which was applied first.

Synergies

When two specific modifiers coexist on the same item, a synergy bonus stat is automatically added. 

deadly:
  synergies:
    sharp: { bonus-stat: CRITICAL_STRIKE_POWER, bonus-value: 5 }

This means: if an item has both "Deadly" and "Sharp", +5 Critical Strike Power is added as a bonus.

Default Synergies

Modifier A Modifier B Bonus Stat Value
Deadly Sharp CRITICAL_STRIKE_POWER +5
Vitality Fortified HEALTH_REGENERATION +1
Blazing Swift FIRE_DAMAGE +2

Stacking Modes

Controls what happens when the same modifier is applied to an item that already has it.

Mode Behavior
REPLACE The new modifier replaces the old one. Old stats are removed, new stats are added.
STACK Both instances coexist. Stats accumulate.
BLOCK Application is refused. "Already applied" message is sent.

Modifier Crafting

Players can combine two identical runes (same modifier ID and tier) to attempt creating a higher tier rune.

How: Drag one rune onto another identical rune in inventory. If both are "Sharp III", the plugin attempts to craft "Sharp IV".

Setting Default Description
crafting.enabled true Toggle crafting on/off
defaults.success-rate 60% Base success rate for all crafting
defaults.fail-destroy-both true Both runes destroyed on fail?

Per-modifier overrides are supported:

overrides:
  sharp:
    1-to-2: { success-rate: 80 }
    2-to-3: { success-rate: 60 }
    3-to-4: { success-rate: 40 }
    4-to-5: { success-rate: 20 }

Scrolls

Protection Scroll

An MMOItem with the NBT stat PROTECTION_SCROLL: true. If the player has one in their inventory when a modifier application fails, the scroll is consumed and the target item is saved.

  • scrolls.protection.auto-consume — automatically consume from inventory (default: true)

Removal Scroll

An MMOItem with the NBT stat REMOVAL_SCROLL: true. Dragging it onto a modified item removes one modifier and reverses its stat changes.

  • scrolls.removal.random-remove: true — removes a random modifier
  • scrolls.removal.random-remove: false — removes the last applied modifier

Tier Overrides

Modifier slot limits can vary based on the MMOItems tier of the target item: 

tier-overrides:
  LEGENDARY: { max: 6, max-prefix: 3, max-suffix: 3 }
  EPIC:      { max: 4, max-prefix: 2, max-suffix: 2 }
  RARE:      { max: 3, max-prefix: 2, max-suffix: 1 }
  UNCOMMON:  { max: 2, max-prefix: 1, max-suffix: 1 }
  COMMON:    { max: 1, max-prefix: 1, max-suffix: 0 }

Items without a tier use the global defaults.

config.yml

Path Default Description
settings.max-modifiers-per-item 6 Global max modifiers per item
settings.max-prefixes 3 Global max prefixes per item
settings.max-suffixes 3 Global max suffixes per item

Animation Config

Each event type (success, critical, fail, protected-fail) has:

  • particle / particle-count — Bukkit Particle name and count
  • sound / sound-volume / sound-pitch — Bukkit Sound name
  • title / subtitle — MiniMessage formatted text (supports {modifier}, {item})
  • title-fade-in / title-stay / title-fade-out — ticks

modifiers.yml

Important This file only defines modifier metadata. Stats are read from the MMOItem rune, not from this file.

Modifier Structure

modifiers:
  sharp:                            # Modifier ID (matches MODIFIER_ID on rune)
    name: "Sharp"                   # Display name
    type: PREFIX                    # PREFIX or SUFFIX
    category: "offensive"           # Organizational category
    tiers:
      1: { success-rate: 95, weight: 1 }
      2: { success-rate: 85, weight: 2 }
      3: { success-rate: 70, weight: 3 }
      4: { success-rate: 50, weight: 4 }
      5: { success-rate: 30, weight: 5 }
    critical-chance: 5              # % chance of critical success
    allowed-types: [SWORD, AXE]    # MMOItems types this can apply to
    conflicts: [dull]              # Cannot coexist with these IDs
    synergies:
      deadly: { bonus-stat: CRITICAL_STRIKE_POWER, bonus-value: 5 }
    stacking: REPLACE               # STACK, REPLACE, or BLOCK

stat-formats.yml

Defines the display format for each individual MMOItems stat. Contains 100+ stat entries organized by category.

Format Placeholders

Placeholder Description
{value} The stat value (absolute, no sign)
<plus> Shows + for positive values, - for negative
{sign} Always shows + or -
{name} Auto-generated display name from stat key

Example Entries

# Weapon Stats
attack-damage: '&3 &7➸ Attack Damage: &f{value}'
critical-strike-chance: '&3 &7■ Crit Strike Chance: &f<plus>{value}%'

# Elemental
fire-damage: '&3 &6🔥&7 Fire Damage: &f<plus>{value}'
ice-damage: '&3 &b❄&7 Ice Damage: &f<plus>{value}'

# Armor
max-health: '&3 &7❤ Health:&c <plus>{value}'

ability:

Commands

All commands require the slmod.admin permission.

Command Description
/slmod reload Reload all configuration files.
/slmod apply <id> <tier> [player] Force-apply a modifier to the held item (bypasses success roll).
/slmod remove <index|id> [player] Remove a modifier from the held item by index or ID.
/slmod info [player] Display all applied modifiers on the held item.
/slmod list List all available modifier IDs and their tier ranges.
Note The apply command is an admin override. It adds the modifier metadata but uses empty stats (no stat bonuses). To properly test, use actual rune items through drag-and-drop.

Permissions

Permission Default Description
slmod.admin op Access to all /slmod commands
slmod.use true Allow drag-and-drop modifier application (when require-permission is enabled)
